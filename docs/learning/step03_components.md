## 1. STM32H743 — 뇌 (제어 MCU)

### 이것이 뭔가?
- ST마이크로일렉트로닉스가 만든 **32비트 마이크로컨트롤러**
- ARM Cortex-M7 코어, 최대 480MHz
- "마이크로컨트롤러" = 작은 컴퓨터. CPU + 메모리 + 입출력이 칩 하나에 들어있음

### 왜 이걸 선택했나? (vs 기존 Teensy 4.1)
| 비교 | Teensy 4.1 | STM32H743 |
|------|-----------|-----------|
| 코어 | Cortex-M7 600MHz | Cortex-M7 480MHz |
| CAN | 소프트웨어 (FlexCAN) | **하드웨어 CAN FD** (네이티브) |
| 양산성 | 취미용 보드 | **산업용, 대량생산 가능** |
| 인증 | 없음 | **의료/산업 인증 가능** |
| 가격 | $30 (보드) | ~$10 (칩만) |
| 듀얼코어 | 없음 | **있음** (M4 보조코어) |

### AR Walker에서의 역할
```
매 2ms (500Hz)마다:
  1. INA828에서 로드셀 값 읽기 (ADC)
  2. EBIMU에서 IMU 데이터 수신 (UART)
  3. 제어 알고리즘 계산 (토크 명령 산출)
  4. Elmo에 토크 명령 전송 (CAN FD)
  5. ESP32에 데이터 전송 (SPI)
  6. Jetson에 데이터 전송 (UART)
```

### 사용하는 핀/기능
| 기능 | 주변장치 | 연결 대상 |
|------|---------|----------|
| FDCAN1/2 | CAN FD | Elmo Gold Twitter |
| SPI1 | SPI 마스터 | ESP32-S3 |
| USART1 | UART | Jetson |
| USART2 | UART | EBIMU RF (수신) |
| ADC1 | 16비트 ADC | INA828 x4 출력 |
| TIM2 | 타이머 인터럽트 | 500Hz 제어 루프 |

## 2. ESP32-S3 — 입과 귀 (통신 MCU)

### 이것이 뭔가?
- Espressif가 만든 Wi-Fi + Bluetooth 내장 마이크로컨트롤러
- 듀얼코어 Xtensa LX7, 최대 240MHz
- **무선 통신 전문가**: Wi-Fi, BLE 5.0, ESP-NOW 모두 지원

### 왜 통신을 따로 분리했나?
```
STM32 혼자서 모터 제어 + Wi-Fi + BLE를 하면:
  → Wi-Fi 패킷 처리 중 모터 제어 루프가 지연될 수 있음
  → 500Hz 실시간 제어에 치명적!

해결: 역할 분리
  → STM32: 실시간 제어에만 집중 (500Hz 보장)
  → ESP32: 통신에만 집중 (지연 OK)
```

### AR Walker에서의 역할
```
STM32로부터 SPI로 데이터 수신
  ↓
세 가지 경로로 데이터 전송:
  ├→ Wi-Fi: PyQt6 GUI에 실시간 데이터 스트리밍
  ├→ BLE 5.0: 스마트폰 모니터링
  └→ ESP-NOW: 추가 센서 노드 통신 (필요 시)
```

### 기존 시스템과 비교
| 비교 | Arduino Nano 33 BLE | ESP32-S3 |
|------|-------------------|----------|
| 무선 | BLE만 | **Wi-Fi + BLE + ESP-NOW** |
| 속도 | 64MHz 싱글코어 | **240MHz 듀얼코어** |
| 메모리 | 256KB | **512KB + PSRAM** |
| MCU 통신 | UART (느림) | **SPI (고속)** |

## 3. Elmo Gold Twitter — 근육 (모터 드라이버)

### 이것이 뭔가?
- **산업용 서보 드라이버** (이스라엘 Elmo Motion Control 제품)
- 48V 전원을 받아서 모터에 정밀한 전류/토크/속도/위치 제어
- 연구실에 **이미 있는 기존 장비**

### 왜 모터 드라이버가 필요한가?
```
MCU(STM32)는 3.3V, 수십 mA밖에 못 줌
모터는 48V, 수십 A가 필요함

MCU ─[작은 신호: "10Nm 토크 줘"]─→ Elmo ─[큰 전력: 48V, 15A]─→ 모터
     (CAN FD 명령)                      (PWM 전력)
```

### MCU와의 통신
- **CAN 버스** 사용 (CANopen 프로토콜)
- STM32에서 CAN FD로 토크/속도/위치 명령 전송
- Elmo에서 현재 위치/속도/전류 피드백 수신

### 기존 시스템과 비교
| 비교 | AK60/AK70/AK80 | Elmo Gold Twitter |
|------|----------------|-------------------|
| 타입 | 모터+드라이버 일체형 | **드라이버만** (모터 별도) |
| 프로토콜 | 독자 CAN 프로토콜 | **CANopen 표준** |
| 제어 모드 | 토크/위치/속도 | 토크/위치/속도 + **더 많은 모드** |
| 파라미터 | 제한적 | **Elmo Studio로 상세 튜닝** |

## 4. INA828 — 촉각 (로드셀 앰프)

### 이것이 뭔가?
- Texas Instruments의 **계측 증폭기 (Instrumentation Amplifier)**
- 아주 작은 전압 신호를 크게 증폭해서 ADC가 읽을 수 있게 함

### 왜 필요한가?
```
로드셀 (힘 센서):
  사람이 워커에 기대는 힘을 측정
  출력: 수 mV (밀리볼트) — 너무 작아서 ADC가 제대로 못 읽음

INA828:
  수 mV → 수 V로 증폭 (게인 설정 가능)
  → STM32 ADC (0~3.3V 범위)가 정확하게 읽을 수 있음
```

### 기존 시스템과 비교
| 비교 | 12비트 ADC 직접 | INA828 + 16비트 ADC |
|------|---------------|-------------------|
| 정밀도 | 12비트 (4096단계) | **16비트 (65536단계)** |
| 노이즈 | 높음 | **낮음** (차동 입력) |
| 분해능 | 0.8mV/step | **0.05mV/step** |

### 회로 개요
```
[로드셀] ──(수 mV)──→ [INA828] ──(0~3.3V)──→ [STM32 ADC]
                       │
                    게인 저항 Rg로
                    증폭률 설정
```

**x4 = 4개 사용** (좌/우 무릎/발목 각 1개)

## 5. EBIMU RF — 균형감각 (IMU)

### 이것이 뭔가?
- **관성 측정 장치 (Inertial Measurement Unit)**
- 가속도계 + 자이로스코프 + 지자기 센서가 합쳐진 것
- RF(무선) 방식으로 데이터 전송 → 배선 불필요
- AHRS (자세 기준 시스템) 내장 → 롤/피치/요 각도 직접 출력

### AR Walker에서의 역할
```
환자의 다리/몸통에 부착
  → 관절 각도, 기울기, 회전 측정
  → 2.4GHz RF로 리시버에 전송 (100Hz)
  → 리시버가 UART로 STM32에 데이터 전달
```

### 스펙
- 출력 데이터율: 100Hz
- 통신: 2.4GHz RF (리시버 → UART)
- 정밀도: 롤/피치 ±0.5°, 요 ±1°

## 6. Jetson Orin Nano Super — 고차원 사고 (AI)

### 이것이 뭔가?
- NVIDIA의 **엣지 AI 컴퓨터**
- 67 TOPS (초당 67조 번 연산) 의 AI 성능
- 리눅스 기반 풀 컴퓨터 (Ubuntu)

### AR Walker에서의 역할
```
STM32에서 UART로 센서 데이터 수신
  → 보행 패턴 분석 (딥러닝)
  → 다음 걸음 예측
  → 적응형 제어 파라미터 계산
  → 결과를 다시 STM32로 전송
```

### 왜 MCU에서 직접 안 하나?
```
STM32: 실시간 제어에 최적화 (500Hz, 결정론적)
       but AI 연산은 느림

Jetson: AI 연산에 최적화 (GPU 가속)
        but 실시간 제어는 부적합 (리눅스 → 비결정론적)

→ 각자 잘하는 것을 맡김!
```

## 7. CAN FD 트랜시버 — 통역사

### 이것이 뭔가?
- MCU의 디지털 CAN 신호를 **CAN 버스의 차동 신호로 변환**
- 예: MCP2542FD, TCAN4550

### 왜 필요한가?
```
STM32 CAN 핀: 0V / 3.3V 디지털 신호 (TX, RX)
CAN 버스:     CAN_H와 CAN_L의 차동 신호 (노이즈에 강함)

트랜시버가 이 둘을 변환해줌:
  STM32 TX → 트랜시버 → CAN_H, CAN_L (차동) → Elmo
  Elmo → CAN_H, CAN_L → 트랜시버 → STM32 RX
```

### CAN 차동 신호란?
```
CAN_H: ──┐  ┌──     두 선의 전압 차이로 데이터 전달
         │  │       → 외부 노이즈가 두 선에 동일하게 영향
CAN_L: ──┘  └──     → 차이는 변하지 않음 → 노이즈에 강함!
```
