# Step 4: 디지털 통신 프로토콜

> STM32, ESP32, Elmo, Jetson이 서로 "대화"하는 방법을 이해하자.

---

## 왜 통신 프로토콜이 중요한가?

AR Walker에는 여러 칩이 있고, 이들이 데이터를 주고받아야 한다:
```
STM32 ←→ Elmo    : "10Nm 토크 줘" / "현재 5.2Nm이야"
STM32 ←→ ESP32   : "센서 데이터 여기 있어" / "GUI에서 명령 왔어"
STM32 ←→ Jetson  : "로드셀+IMU 데이터야" / "보행 예측 결과야"
STM32 ← EBIMU    : "현재 각도 15.3도야"
```

각 연결마다 **적합한 통신 방식**이 다르다.

---

## 1. UART (Universal Asynchronous Receiver/Transmitter)

### 한 줄 요약
> 가장 간단한 시리얼 통신. 선 2개로 양방향 대화.

### AR Walker에서 사용
- **STM32 ↔ Jetson** (양방향 데이터)
- **EBIMU RF 리시버 → STM32** (IMU 데이터 수신)

### 동작 원리
```
장치 A                    장치 B
  TX ──────────────────→ RX     (A가 B에게 보냄)
  RX ←────────────────── TX     (B가 A에게 보냄)
  GND ─────────────────── GND   (공통 기준)
```

### 핵심 특징
| 특징 | 내용 |
|------|------|
| 핀 수 | **2개** (TX, RX) + GND |
| 동기/비동기 | **비동기** (클럭 선 없음) |
| 속도 | 보통 9600 ~ 115200 bps, 최대 수 Mbps |
| 연결 구조 | **1:1** (점대점) |
| 방향 | **양방향** (Full Duplex) |

### 보드레이트 (Baud Rate)
- 양쪽 장치가 **같은 속도로 설정**해야 통신 가능
- 시계가 없으므로, 속도를 미리 약속해야 함
```
STM32 UART: 115200 bps
Jetson UART: 115200 bps   ← 반드시 동일!
```

### 데이터 형식
```
[시작 비트(0)] [데이터 8비트] [패리티(선택)] [정지 비트(1)]
     1bit          8bit          0-1bit        1-2bit

예: 문자 'A' (0x41 = 01000001) 전송
  0 | 1 0 0 0 0 0 1 0 | 1
  시작   데이터(LSB first)  정지
```

### 장단점
```
장점: 간단함, 핀 적음, 대부분의 MCU에 내장
단점: 1:1만 가능, 속도 제한, 거리 짧음
```

## 2. SPI (Serial Peripheral Interface)

### 한 줄 요약
> 고속 동기 통신. 클럭이 있어서 빠르고 정확하다.

### AR Walker에서 사용
- **STM32 (마스터) ↔ ESP32-S3 (슬레이브)**

### 동작 원리
```
STM32 (마스터)              ESP32 (슬레이브)
  SCK  ──────────────────→ SCK    (클럭 - 마스터가 생성)
  MOSI ──────────────────→ MOSI   (Master Out, Slave In)
  MISO ←────────────────── MISO   (Master In, Slave Out)
  CS   ──────────────────→ CS     (Chip Select - 슬레이브 선택)
  GND  ──────────────────── GND
```

### 핵심 특징
| 특징 | 내용 |
|------|------|
| 핀 수 | **4개** (SCK, MOSI, MISO, CS) + GND |
| 동기/비동기 | **동기** (클럭 있음) |
| 속도 | **수십 MHz** (UART보다 훨씬 빠름) |
| 연결 구조 | **1:N** (마스터 1개, 슬레이브 여러 개 가능) |
| 방향 | **양방향 동시** (Full Duplex) |

### 왜 STM32↔ESP32에 SPI를 쓰나?
```
UART:  최대 수 Mbps → 500Hz × 큰 데이터 패킷이면 부족할 수 있음
SPI:   수십 MHz    → 충분히 빠름, 실시간 제어에 영향 없음
I2C:   수백 kbps   → 너무 느림
```

### 마스터/슬레이브 관계
```
마스터 (STM32): 클럭을 생성하고, 언제 통신할지 결정
슬레이브 (ESP32): 마스터가 시키면 응답

→ STM32가 제어 루프의 주인이므로 마스터
→ ESP32는 데이터를 받아서 전송만 하므로 슬레이브
```

### 장단점
```
장점: 매우 빠름, 동시 양방향, 단순한 프로토콜
단점: 핀이 많음 (4개+), 거리 짧음 (같은 보드 내)
```

## 3. CAN / CAN FD (Controller Area Network)

### 한 줄 요약
> 자동차/산업용 표준. 노이즈에 강하고, 여러 장치가 한 버스를 공유.

### AR Walker에서 사용
- **STM32 ↔ Elmo Gold Twitter** (모터 제어 명령/피드백)

### 동작 원리
```
    CAN 버스 (꼬인 2선: CAN_H, CAN_L)
  ═══════════════════════════════════════
  │           │            │           │
[STM32]    [Elmo 1]    [Elmo 2]    [Elmo 3]
+트랜시버  (내장)       (내장)      (내장)
  │                                    │
[120Ω]                              [120Ω]  ← 종단 저항
```

### 핵심 특징
| 특징 | 내용 |
|------|------|
| 핀 수 | **2개** (CAN_H, CAN_L) — 차동 신호 |
| 동기/비동기 | **동기** (비트 동기화 내장) |
| 속도 | CAN: 1 Mbps, **CAN FD: 최대 8 Mbps** |
| 연결 구조 | **버스형** (여러 장치가 한 선을 공유) |
| 방향 | **양방향** (Half Duplex — 한 번에 하나만 전송) |

### CAN vs CAN FD
```
CAN (Classic):
  - 최대 1 Mbps
  - 데이터: 최대 8바이트/프레임
  - 20년 이상 검증된 표준

CAN FD (Flexible Data-rate):
  - 제어부: 1 Mbps (호환성)
  - 데이터부: 최대 8 Mbps (빠름!)
  - 데이터: 최대 64바이트/프레임 (8배!)

→ AR Walker는 CAN FD 사용 → 더 많은 데이터를 더 빠르게
```

### 차동 신호가 노이즈에 강한 이유
```
일반 신호:          차동 신호 (CAN):
  신호선 ──→          CAN_H ──→
                      CAN_L ──→

노이즈 발생 시:
  신호선: 3.3V + 0.5V(노이즈) = 3.8V  → 오류!

  CAN_H: 3.5V + 0.5V = 4.0V
  CAN_L: 1.5V + 0.5V = 2.0V
  차이:  4.0 - 2.0 = 2.0V              → 원래와 같음! 노이즈 무시!
```

### CAN 메시지 구조
```
[SOF] [ID (11/29bit)] [제어] [데이터 (0~64 byte)] [CRC] [ACK] [EOF]

ID:    메시지 종류 + 우선순위 (숫자 낮을수록 우선)
데이터: 실제 전송할 값 (토크 명령, 센서 값 등)
CRC:   오류 검출 코드
ACK:   수신자가 "잘 받았다" 확인
```

### 종단 저항 (120Ω)
```
CAN 버스의 양 끝에 120Ω 저항을 달아야 함
→ 신호 반사를 방지 (반사되면 신호가 깨짐)
→ 양쪽 합쳐서 60Ω
```

### Elmo와의 통신 예시
```
STM32 → Elmo (토크 명령):
  ID: 0x601 (Elmo Node 1의 SDO 수신)
  데이터: [토크값 4바이트] + [명령 코드]

Elmo → STM32 (피드백):
  ID: 0x581 (Elmo Node 1의 SDO 응답)
  데이터: [현재 위치] [현재 속도] [현재 전류]
```

### 장단점
```
장점: 노이즈에 매우 강함, 버스형 (여러 장치), 오류 검출 내장, 산업 표준
단점: 속도 제한 (SPI보다 느림), 트랜시버 필요
```

## 4. I2C (Inter-Integrated Circuit)

### 한 줄 요약
> 핀 2개로 여러 센서를 연결. 느리지만 간단.

### AR Walker에서 사용
- 현재 직접 사용하지 않지만, 일부 센서 확장 시 사용 가능

### 동작 원리
```
       I2C 버스
  SDA ═══════════════════   (데이터)
  SCL ═══════════════════   (클럭)
  │       │       │
[마스터] [센서1]  [센서2]
(STM32)  (0x48)   (0x69)   ← 각 장치의 고유 주소
```

### 핵심 특징
| 특징 | 내용 |
|------|------|
| 핀 수 | **2개** (SDA, SCL) + GND |
| 속도 | 100kHz ~ 3.4MHz (느림) |
| 연결 구조 | **버스형** (주소로 구분) |
| 방향 | **양방향** (Half Duplex) |

### 왜 AR Walker에서 주력으로 안 쓰나?
```
I2C: 느림 (최대 3.4MHz) → 500Hz 제어에 병목
SPI: 빠름 (수십 MHz) → 실시간 데이터 교환에 적합
CAN: 노이즈에 강함 → 모터 통신에 적합
```

```
                          ┌──────────────────┐
                          │   Jetson Orin     │
                          │   Nano Super      │
                          └────────┬──────────┘
                                   │
                              UART │ 115200~921600 bps
                              (2선) │ TX, RX
                                   │
┌───────────────┐    SPI     ┌─────┴──────────┐
│   STM32H743   │◄──────────►│   ESP32-S3     │
│   (마스터)    │  수십 MHz  │   (슬레이브)   │
│               │   (4선)    │               │
└──┬─────┬──────┘   MOSI,   └──┬────┬───┬───┘
   │     │          MISO,       │    │   │
   │     │          SCK, CS     │    │   │
   │     │                      │    │   │
   │     │ UART (수신)    Wi-Fi │  BLE │ ESP-NOW
   │     │ (1선)                │    │   │
   │     │                      │    │   │
   │  [EBIMU RF]          [PC/GUI] [폰] [센서]
   │  (IMU 리시버)
   │
   │ CAN FD
   │ (2선: CAN_H, CAN_L)
   │ 1~8 Mbps
   │
   ├──→ [Elmo 1] → [모터 1]
   ├──→ [Elmo 2] → [모터 2]
   ├──→ [Elmo 3] → [모터 3]
   └──→ [Elmo 4] → [모터 4]
```

## 프로토콜 비교 총정리

| 프로토콜 | 핀 수 | 속도 | 토폴로지 | AR Walker 용도 |
|---------|-------|------|---------|---------------|
| **UART** | 2 | ~1 Mbps | 1:1 | Jetson, EBIMU |
| **SPI** | 4 | ~50 MHz | 1:N | STM32↔ESP32 |
| **CAN FD** | 2 | ~8 Mbps | 버스 | STM32↔Elmo 모터 |
| **I2C** | 2 | ~3.4 MHz | 버스 | (예비/확장용) |
| **Wi-Fi** | 0 (무선) | ~50 Mbps | 스타 | ESP32↔GUI |
| **BLE 5.0** | 0 (무선) | ~2 Mbps | 스타 | ESP32↔스마트폰 |
| **ESP-NOW** | 0 (무선) | ~1 Mbps | P2P | ESP32↔센서노드 |

## 연습 문제

### 문제 1: 프로토콜 선택
> STM32에서 Elmo 모터 드라이버 4개에 토크 명령을 보내려면 어떤 프로토콜이 적합한가? 그 이유는?

<details>
<summary>정답 보기</summary>

```
CAN FD가 적합

이유:
1. 버스형 → 선 2개로 4개 Elmo 모두 연결 가능
2. 차동 신호 → 모터 노이즈에 강함
3. CAN FD → 8Mbps, 64바이트로 충분한 속도/데이터
4. Elmo가 CAN(CANopen)을 기본 지원
5. 산업 표준 → 신뢰성 검증됨
```
</details>

### 문제 2: 핀 수 계산
> STM32H743에서 다음 모든 통신을 위해 필요한 최소 핀 수는?
> - ESP32와 SPI 통신
> - Jetson과 UART 통신
> - Elmo와 CAN FD 통신
> - EBIMU에서 UART 수신

<details>
<summary>정답 보기</summary>

```
SPI (ESP32):    4핀 (SCK, MOSI, MISO, CS)
UART (Jetson):  2핀 (TX, RX)
CAN FD (Elmo):  2핀 (CAN_TX, CAN_RX) → 트랜시버 거침
UART (EBIMU):   1핀 (RX만, 수신 전용)

합계: 4 + 2 + 2 + 1 = 9핀

(+ ADC 핀 4개 for INA828, GPIO 등은 별도)
```
</details>

### 문제 3: 데이터 전송 시간
> 500Hz 제어 루프에서 매 사이클(2ms)마다 Elmo 4개에 각 8바이트 명령을 보내고,
> 각 8바이트 응답을 받아야 한다. CAN FD (8Mbps)에서 가능한가?

<details>
<summary>정답 보기</summary>

```
전송 데이터: 4 × 8바이트 × 2(송수신) = 64바이트 = 512비트
CAN FD 오버헤드 포함: 약 512 × 2 = ~1024비트 (프레임 헤더, CRC 등)
전송 시간: 1024비트 / 8,000,000 bps = 0.128ms

2ms 중 0.128ms만 사용 → 6.4% 사용률
→ 충분히 가능! 여유 93.6%
```
</details>

## 핵심 요약

**AR Walker 통신 규칙:**
1. **모터 제어** (실시간, 노이즈 많음) → **CAN FD**
2. **MCU 간 고속 데이터** (같은 보드) → **SPI**
3. **AI 컴퓨터 연결** (범용, 간단) → **UART**
4. **무선 모니터링** (GUI, 원격) → **Wi-Fi / BLE**
